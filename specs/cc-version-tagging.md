# CC Revision Spec: Git Hash Version Tagging

**Date:** 2026-03-20
**Severity:** Low (metadata-only change, zero functional risk)
**Scope:** `src/supabase.ts`, new `src/version.ts`, new `deploy.sh`, new `VERSION_REGISTRY.md`, `.gitignore`

---

## What Went Wrong

The worker hardcodes `worker_version: 'v2'` in all 5 Supabase table writes: `audit_logs`, `leads_audit_logs`, `run_summaries`, `daily_snapshots`, `notifications`. The tag was meant to distinguish v1 (MCP-only) from v2 (direct API), but it has never changed across multiple subsequent deploys with materially different behavior.

Every row written between 2026-03-18 and 2026-03-20 carries the same `'v2'` tag. A row from a broken March 18 deploy (Supabase writes dropped, 0-indexed steps) and a row from a healthy March 20 deploy (writes working, 1-indexed steps) are indistinguishable at query time. Agents building on or auditing CC data cannot tell which code produced a given row without manually cross-referencing timestamps against a mental changelog.

There is also no clear trigger for when to bump to `'v3'`. That ambiguity will compound as the system continues to evolve.

---

## Decision

Use the **git short hash** as the version tag, auto-generated at build time.

- Ties every data row to exact code via `git show <hash>`
- Requires zero manual bumping
- Available at build time (CF Worker Version ID is not accessible at runtime during cron triggers)
- Compatible with the existing `text` column type in all 5 tables

### 3. Create `builds/auto-turn-off/deploy.sh`

```bash
#!/bin/bash
set -e

# Generate version tag from current git short hash
HASH=$(git rev-parse --short HEAD)
echo "// Auto-generated by deploy.sh. Do not edit manually." > src/version.ts
echo "export const WORKER_VERSION = '${HASH}';" >> src/version.ts

echo "[deploy] Worker version: ${HASH}"
npx wrangler deploy
```

Make executable: `chmod +x deploy.sh`

All future deploys use this script. Raw `npx wrangler deploy` must not be used.

## What NOT to Change

- KV key patterns -- version is not part of any KV key
- KV log entry values -- traceability is via Supabase, not KV

- Dedup keys -- completely unrelated

## Migration

No Supabase schema changes. The `worker_version` column is already `text` in all 5 tables.

Existing rows keep their `'v2'` or `NULL` tags. The VERSION_REGISTRY documents what those mean. No backfill needed.

No KV changes needed.

---

## Risk Assessment

**Zero functional risk.** The version tag is metadata-only. No logic reads `worker_version`.

**Deploy process change.** All future deploys must go through `deploy.sh`. Failure modes:
- `src/version.ts` exists from prior deploy but stale: wrong hash written, functionally harmless
- `src/version.ts` absent (fresh checkout): TypeScript fails, deployment stops, no bad data written

Both are detectable and non-catastrophic.
