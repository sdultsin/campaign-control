## Issues Found

### Critical
1. **Instantly API data integrity** — `get_step_analytics` returns different sent counts and opportunity attribution than the Instantly UI. Discrepancies go in both directions (API under by 63%, over by 16%). Opportunity misattribution: API attributes opps to wrong variants or leaves them in `step: null` rows. **This is a platform issue requiring escalation to Instantly CTO.** Kills paused until resolved.

### High
2. **Deploy hot-swap mid-run** — Cron fired at 11:00 with old code, deploy at 11:03 swapped mid-run. Some workspaces ran dry, others live. No clean way to prevent this. Mitigation: deploy between cron runs, or add a "deploy lock" that waits for current run to finish.
3. **No blocked notification dedup at launch** — 30+ blocked notifications flooded Ido's channel. Fix: added KV dedup. Lesson: every notification type needs dedup from day 1.
4. **Slack rate limiting** — 46 messages in 26 seconds. Fix: 1.5s delay between messages. Thread replies dropped for 14 notifications. Manually backfilled.

### Medium
5. **No dedup keys from first run** — First run sent notifications without writing dedup keys (old code). Required manual seeding of 169 keys into KV.
6. **CM parser missed non-parenthetical names** — "- Alex" format not caught. Fix: fallback parser for known CM names.
7. **Equinox workspace edge case** — Dedicated workspace defaultCm overrode parenthetical name. Fix: check parenthetical first, fall back to defaultCm.

### Low
8. **Run duration** — 15 min base + Slack delays approaching 20 min. Monitor as notification volume grows.
9. **Campaign naming inconsistency** — CMs use inconsistent formats. Need enforced naming convention.
10. **KV query speed** — Each key fetch takes 2-3s. Auditing 300+ entries takes 10-30 min. Supabase sync spec written to solve this.

## Mistakes Made (Lessons)

1. **Triggered second run prematurely** — Should have deployed blocked dedup first, then triggered.
2. **Launched without blocked dedup** — Only had warning dedup and leads dedup. Should have had all notification types deduped before going live.
3. **No pre-launch API data validation** — Should have compared API data to UI data for a sample of campaigns before trusting the kill decisions. The Instantly API discrepancy was discoverable pre-launch.
4. **Deployed during active cron** — 11:03 deploy overlapped with 11:00 cron. Should deploy at :05-:55, never at :00.
